## 1. アプリ本体の制作（観戦HQ '26 として誕生）

### 1-1. コンセプト
「寝不足上等。全104試合、見逃すな。」をスローガンに、**日本のファンがW杯2026を日本時間で完全攻略できるオールインワン観戦アプリ**。プレミアムなUI（ダークネイビー×ゴールド×サムライブルー、Antonフォント、リキッドグラス、オーロラ背景、パーティクルヒーロー）を `/premium-site` スキルで構築。

### 1-2. 技術スタック
- Vite + React 19 + TypeScript + Tailwind CSS 3 / lucide-react
- **完全静的**（サーバー・DB不要）、localStorageで個人データ永続化
- ポート **5180**（5173は別プロジェクト seoul-trip-guide が使用中のため）

### 1-3. 7タブ構成
| タブ | 内容 |
|---|---|
| ホーム | 日本戦カウントダウン、今日/明日の試合、大会10大ストーリー、死の組 |
| 試合日程 | 全104試合(JST)。節/グループ/チーム/放送区分/フィルタ。Googleカレンダー1クリック追加 |
| 日本代表 | 速報・26人メンバー・対戦相手スカウティング・突破シナリオ・F組順位表 |
| 組&チーム | 12組順位表(結果から自動計算)＋全48カ国図鑑 |
| ランキング | 得点/アシストランキング、得点王オッズ、優勝オッズ、懸かる記録 |
| 視聴ガイド | DAZN無料/独占/地上波の整理、プラン料金、無料視聴4手段 |
| マイ観戦 | 推しチーム、勝敗予想と的中率、観戦予定＋睡眠プランナー、結果入力 |

### 1-4. データ層（後の自動化で重要になる土台）
- `src/data/types.ts`: `Match`, `Team`, `MatchResult{matchId,homeScore,awayScore,note?}`, `ScorerEntry{player,team,goals,assists,club?}` 等
- `src/data/matches.ts`: 全104試合。グループ72(id 1-72)＋決勝T 32(id 73-104、home/away="未定")。`m()`/`k()` ヘルパーで生成
- `src/data/teams.ts`: 48カ国（FIFA3文字コード＋iso2）。`flagUrl()` は flagcdn.com（Windowsは絵文字国旗が表示されないため画像化）
- `src/data/results.ts`: `officialResults[]` / `officialScorers[]`（当初は空）
- `src/lib/standings.ts`: `computeStandings()` が結果から順位を計算、`mergeResults()` は **official > user** 優先
- `src/state/AppState.tsx`: React Context。当初はビルド時importのみ、ランタイムfetchは無し

### 1-5. 制作のやり方（効率化の工夫）
- **サブエージェント7体並列**でリサーチ（大会構造・日本代表・各国・放送区分・ストーリー）→ メイン文脈を肥大させない
- **ファクトチェック専任エージェント**で矛盾検証。例: 遠藤航の離脱情報がエージェント間で食い違い → 7体目で「6/11発表・代表引退表明」を確定、`japan.ts` を修正
- **GPT Image 2（codex exec経由）でヒーロー画像生成**。`service_tier` の罠（priorityでパース死・flexはAPI拒否）を踏み、`-c 'service_tier="fast"'` で成功

## 3. 仕組みの理解（静的・localStorage＝端末ごと独立）

ユーザーから「これ、他の人に共有してもそれぞれ使える?」という問い。整理した結論:
- **共有データ**（日程・放送区分・順位表ロジック・図鑑）= アプリに同梱、全員同じ
- **個人データ**（推し・観戦予定・予想・睡眠プラン・結果入力）= 各自の **localStorage（ブラウザ内のメモ帳）** に保存 → **端末ごとに完全独立**、他人に漏れない・混ざらない。逆に自分のスマホ⇔PCも同期しない
- これを「皆で予想を競う」「端末間同期」にするにはバックエンドが必要。今回は不要なのでやらない（完全静的を維持）

→ つまり**URLを配るだけで各自が自分専用の観戦HQとして使える**。

## 4. LINEキーワード配布（「サッカー」→自動でアプリURL）

### 4-1. ねらいと料金の核心
プレゼント企画の定番導線（フォロー＋いいね＋RT＋LINEでキーワード送信）を実装。LINE公式の自動返信には2種類:
- **メッセージ配信(一斉送信)** = 無料枠 月200通まで（有料化の対象）
- **応答メッセージ(キーワード応答・あいさつ)** = **配信数にカウントされず、無料・無制限**

→ 「キーワードでURL配布」は**何人来てもタダ**。アカウント: なりタイ園長｜京大卒AIオタク（**@338kaiex**）。キーワードは **「サッカー」**。

### 4-2. Kimi WebBridgeでブラウザ自動操作（ハマりどころの宝庫）
LINE公式アカウント管理画面（manager.line.biz）をKimi WebBridge（ローカルデーモン:10086）で自動操作。判明した重要ノウハウ:
- **kimiの `fill` は日本語入力を文字化けさせる**（「サッカー」→U+FFFD混じりの mojibake）。→ **`evaluate` でJSから直接値をセット**すれば正確に入る
- 管理画面は **Vue.js + Lit製Webコンポーネント** の混成。本文エディタは `RICH-TEXTAREA` のShadow DOM内 `<textarea>`（CSSセレクタで届かない→shadow root再帰探索）。値セット後は **host要素に input/change を composed:true で発火**させるとVueが認識
- キーワードタグ追加は **Keywordコンポーネント(Vue)の `pushKeyword()` を直呼び**が確実

### 4-3. 24時間発火させる設定
初期は「応答時間内=手動チャット / 時間外=応答メッセージ」で**日中はキーワードが発火しない**問題。
→ 設定 > 応答設定 > チャットの応答方法 > 応答時間内を **「手動チャット＋応答メッセージ」** に変更。これで**24時間**発火、かつ**手動チャットも維持**。

### 4-4. Default(一律応答)の誤爆問題
一律応答は「キーワード以外の**全メッセージ**」に返すので、普通の会話にも定型文をロボ返信してしまう＝事故。ユーザーが鋭く気づいた。
→ **Defaultはオフ**が正解。「サッカー」→URL（キーワード優先）、それ以外→自動返信なしでチャットに入る（手動対応）。打ち間違い対策は「あいさつメッセージで案内」＋「表記ゆれキーワード追加」で代替。

> ここで私が「利用停止押して」→「やっぱりキャンセルで」と指示を二転三転させ、ユーザーを混乱させる失敗。結果はオフ(=正解)に着地。素直に謝罪。

### 4-5. あいさつメッセージ（期間限定CTA）
友だち追加直後に出るメッセージに「『サッカー』とだけ送ってね」のCTAを**差し込み**（原文は区切り線の下に残して復元しやすく）。原文タグ「友だちの表示名」を壊さないため**全置換せず挿入**。

→ **テスト成功**（ユーザーのプライベートLINEで「サッカー」送信→URL着弾を確認）。

## 7. 【本丸】結果・順位・得点・アシスト・ニュースの完全自動更新

「大会始まってるのに結果が全く更新されない。手動入力なんてありえない。全自動にして」というご要望。ここが今回最大の山場。

### 7-1. 多角的リサーチ（ワークフローで設計を競わせる）
プランモードで、3方向を並列調査:
1. コードのデータ層（自動更新が書き込むべき型）
2. データソース（W杯2026の結果・**アシスト**・順位を取れる無料API）
3. 更新の仕組み（静的サイト×Vercelで最効率の自動更新＋ニュース）

さらに **4エージェントのワークフロー**で、競合する3アーキテクチャ案を作って相互に潰し合い（レート制限の試算・ID対応・決勝T自動反映・障害時フォールバック）、最良案を統合。

### 7-2. 採用アーキテクチャ
```
GitHub Actions cron（15分ごと・公開リポジトリ=無料無制限・キーはSecret）
  → scripts/fetch-data.mjs（結果・得点）＋ scripts/rss.mjs（ゲキサカRSS）
  → app形状JSON(MatchResult[]/ScorerEntry[]/NewsItem[])を「data」ブランチへ公開
       ↓ 再デプロイ不要
React: 起動時+可視化時に raw.githubusercontent の data ブランチを取得
  → 既存 mergeResults(live ?? official, user) → computeStandings が順位計算
  → 取得失敗時: raw → Vercel /data → ビルド時static の順でフォールバック。手動入力も温存
```
- サーバーレス+エッジ案は「エッジキャッシュが地域別カウントで無料枠超過」と試算で確認し**不採用**
- Vercel無料Cronは「1日1回」しか不可で**失格**。**GitHub Actions(公開リポジトリ無料無制限)** を採用

### 7-3. Step 0（事前検証）で地雷を踏み抜く
プラン通り「キー取得後にまず本当に取れるか検証」。Kimiで一緒にAPI-Football（api-sports.io）の無料キーを取得 → 検証 → **「Free plans do not have access to this season（2026非対応・2022〜2024のみ）」**。
**API-Football無料はW杯2026に非対応**（現行シーズンは有料$19/月のみ）と判明。事前検証のおかげで無駄な実装を回避。

→ ユーザー判断で **完全無料を維持** → **football-data.org（無料）に切替**。再度Kimiで無料トークンを登録・取得し検証 → **W杯2026（season 2026・104試合・既に複数消化）が無料で取得OK**。

### 7-4. 嬉しい誤算（アシスト）
「football-data無料はアシスト取れない」想定だったが、`/competitions/WC/scorers?limit=100` で**得点上位選手のアシストが一部取れる**ことが判明。アシストランキングが空ではなく実データで埋まった（Undav 2、Isak 2 等）。

### 7-5. マッピング（ここが効率化の鍵）
- **チームコード**: football-dataのTLAがアプリのFIFAコードと**ほぼ完全一致**。48カ国中**差分は `URУ→URU`（ウルグアイ）の1個だけ**。複雑な「team-id対応表」は不要、エイリアス1個で解決（`fetch-data.mjs` の `ALIAS={URY:'URU'}`）
- **fixture→matchId**: `scripts/maps/schedule.json`（matches.tsから正規表現で生成・104件）を **(JST日付 | home | away)** で突合。home/away反転時はスコアも入替
- **UTC→JST**: 日本はDST無し＝固定+9h。04:00 JSTキックオフ=前日UTC、というズレに注意して日付キーを導出
- **決勝T(73-104)**: 日付・会場は確定済み。6/28のブラケット確定後に各stage内でkickoff順に positional 反映（**未実装＝後日**。スコアはJSONで即ライブ可、チーム名ラベルのみ matches.ts 1回コミット）

## 8. 技術的な学び・ハマりどころ集（再利用ナレッジ）

1. **Kimi WebBridge × 日本語入力**: `fill` は文字化け → `evaluate` で直接セット。Vue/Lit製はhost要素にcomposedイベント発火、`pushKeyword()` 等のコンポーネントメソッド直呼びが確実
2. **LINE公式の料金**: 応答メッセージ(キーワード/あいさつ)は無料・無制限。課金対象は一斉配信のみ
3. **LINE一律応答の誤爆**: 普通の会話にロボ返信する → オフが基本。打ち間違いはあいさつ＋表記ゆれで救済
4. **API-Football無料はWC2026非対応**（2022-2024のみ）。現行大会は有料。→ **football-data.org無料**が代替（10回/分・1日上限なし・WCは無料コンペ）。ただしスコアはややディレイ
5. **静的サイトの自動更新**: Vercel無料Cronは1日1回で不可 → **GitHub Actions(公開リポジトリ無料)** が正解。`data`ブランチ＋raw.githubusercontent(CORS:*・5分CDN)で**再デプロイ不要**の更新
6. **JamesIves/github-pages-deploy-action の clean:false** = 今回出力に無いファイルは消さない＝失敗系統の前回正データを温存（last-good）
7. **GCMトークンのディスク書き込みはガードで拒否**される。秘密設定はブラウザUI or API一時利用で
8. **GitHub Secret画面**: `textarea[name="secret_value"]` を正しく指定（最初のtextareaは隠しfeedback欄）
9. **マッピングは"データ"であって"転送"ではない**: どのアーキテクチャでも同じ。TLA一致のおかげでエイリアス1個で済んだ
10. **事前検証(Step 0)は命**: API-Football無料の地雷を本実装前に踏み抜けたのは検証ゲートのおかげ

## 10. 今後のメンテナンス

- **結果が更新されない時**: GitHub Actions の `refresh-data` 実行ログと、`data`ブランチの `meta.json`（degraded/unmapped）を確認
- **トークン失効時**: GitHub Secret `FOOTBALL_DATA_TOKEN` を再設定（football-data.orgで再発行）
- **決勝トーナメント(6/28以降)**: ブラケット確定後、`fetch-data.mjs` に positional 反映を追加実装。チーム名ラベルは `matches.ts` を1回コミットして "未定"→実コードに置換（スコアは今もJSON経由で反映可）
- **手動上書き**: 緊急時は `src/data/results.ts` の `officialResults`/`officialScorers` に追記すれば、`mergeResults` の official優先でライブ値を上書きできる
- **配布企画の設定戻し**: 終了後、LINEのあいさつメッセージCTA削除・Default応答の扱いを元に戻す

## 11. 一連の流れ・総括

ゼロからのWebアプリ制作 → 公開 → 仕組みの理解 → LINEキーワード配布 → リブランド → 告知文 → **完全自動更新**まで、**1本の流れ**で走り切った。

最大の収穫は2つ:
1. **LINEキーワード自動返信が無料・簡単に作れる**と体感したこと（あらゆる配布企画に再利用可能）
2. **「手動はありえん、絶対自動化できる」**という発想から、無料APIの地雷を踏み抜きつつ、完全無料の全自動データパイプラインを本番稼働まで持っていったこと

これで「ワールドカップ手帳 '26」は、**観るたびに最新**の、本当の意味での"手帳"になった。
